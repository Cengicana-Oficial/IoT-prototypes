1. ¿Qué es este proyecto?

PV-Davies01 es un pluviómetro inteligente — un dispositivo que mide cuánta lluvia cae, acumula los datos cada minuto, y los transmite de forma inalámbrica a través de LoRaWAN hacia un servidor central (ChirpStack en CENGICANA). No necesita WiFi, no necesita cables de red. Usa ondas de radio de largo alcance y ultra bajo consumo.

💡 Dato clave Este dispositivo puede operar por meses o años con una simple batería, transmitiendo datos a kilómetros de distancia. Es la magia de LoRaWAN.

El problema que resuelve

En agricultura e investigación climática (como en CENGICANA — Centro Guatemalteco de Investigación y Capacitación de la Caña de Azúcar), necesitás saber exactamente cuánta lluvia cayó en cada zona, en tiempo real, sin ir físicamente a vaciar un balde medidor. Este dispositivo automatiza todo: mide, acumula, transmite.

Los tres protagonistas

BastWAN v2.0Placa Arduino-compatible con módulo LoRa incorporado (RFM95). Lee los pulsos y transmite por radio.

2. Arquitectura del sistema

Así viajan los datos desde la gota de lluvia hasta el servidor:

📡 El viaje completo de un dato Gota → balancín bascula → imán pasa por reed switch → pulso eléctrico en pin 9 del BastWAN → Arduino cuenta el pulso → cada 60 segundos codifica el acumulado en 2 bytes → LoRa transmite a 868/915 MHz → Gateway LoRaWAN recibe la señal → Gateway reenvía por Internet (TCP/IP) → ChirpStack decodifica el payload → base de datos. Todo en menos de 2 segundos.

3. Hardware: Componentes físicos

Davis 7345.440 — El sensor de lluvia

Es un pluviómetro de balancín (tipping bucket rain gauge). Dentro tiene un embudo que canaliza el agua hacia un pequeño balancín con dos cubos. Cuando un cubo acumula exactamente 0.2 mm de lluvia, el peso lo hace bascular. Un imán en el balancín pasa frente a un reed switch (interruptor magnético), cerrando un contacto eléctrico por una fracción de segundo. Eso es un pulso.

💡 ¿Por qué 0.2 mm? Es el estándar métrico para pluviómetros Davis. En pulgadas sería 0.01". Cada pulso = 0.2 mm de precipitación. Para ChirpStack, que espera unidades de 0.1 mm, cada pulso vale 2 unidades.

BastWAN v2.0 — El cerebro con radio

BastWAN es una placa de desarrollo creada por Electronic Cats (México). Combina un microcontrolador SAMD21G18 (ARM Cortex-M0+ a 48 MHz, el mismo chip que un Arduino Zero) con un módulo RFM95 de HopeRF para comunicación LoRa en bandas de 868/915 MHz.

Conexión física

El cable del Davis 7345.440 tiene dos hilos. Uno va a GND (tierra) y el otro al pin digital 9 del BastWAN. El pin se configura como INPUT_PULLUP, lo que significa que internamente se conecta a VCC (3.3V) a través de una resistencia. Cuando el reed switch se cierra, el pin va a GND (LOW). Cuando está abierto, la resistencia pull-up lo mantiene en HIGH.

4. LoRaWAN: La red que todo lo conecta

¿Qué es LoRa?

LoRa (Long Range) es una técnica de modulación de radiofrecuencia patentada por Semtech. Usa chirp spread spectrum (CSS) — "chirridos" de frecuencia que se expanden en el espectro. Esto le da dos superpoderes:

  • Alcance extremo: 2-5 km en ciudad, 10-20 km en campo abierto.
  • Penetración: Atraviesa paredes, vegetación, y obstáculos mejor que WiFi o Bluetooth.
  • Ultra bajo consumo: Un sensor puede durar años con una batería AA.

¿Qué es LoRaWAN?

LoRaWAN es el protocolo de red que va encima de LoRa. Define:

  • Cómo los dispositivos se autentican y se unen a la red.
  • Cuándo y cómo transmiten (clases A, B, C).
  • Cómo se encriptan los datos (AES-128).
  • Cómo los gateways reenvían paquetes al network server.

Clases de dispositivos LoRaWAN

ClaseComportamientoConsumoUso típico
CLASS A Transmite cuando quiere. Después de cada uplink, abre DOS ventanas cortas de recepción para downlinks. El resto del tiempo DUERME. Mínimo Sensores a batería (nuestro pluviómetro)
CLASS B Como Class A, pero además abre ventanas de recepción programadas (beacons) para recibir downlinks en horarios fijos. Medio Actuadores que necesitan recibir comandos
CLASS C Escucha CONTINUAMENTE. Solo deja de escuchar mientras transmite. Alto (requiere alimentación permanente) Dispositivos conectados a la red eléctrica

Nuestro pluviómetro usa CLASS A — la más eficiente. Transmite 1 vez por minuto, escucha unos milisegundos por si el servidor quiere responder, y vuelve a dormir.

Spreading Factor (SF) y Data Rate

El código configura SF8BW125. Vamos a desglosarlo:

  • SF = Spreading Factor (8): Cada bit de datos se "expande" en 28 = 256 chips. A mayor SF, mayor alcance pero menor velocidad. SF7 es lo más rápido; SF12 lo más lento pero de mayor alcance. SF8 es un balance: buen alcance (~5 km) con velocidad decente.
  • BW = Bandwidth (125 kHz): El ancho de banda del canal. 125 kHz es el estándar más común para LoRaWAN.
📡 SF8BW125 en números Data rate: ~3,125 bps. Tiempo de transmisión para 2 bytes de payload: ~50 ms. Alcance típico: 3-8 km (dependiendo de obstáculos). Perfecto para un pluviómetro en campo agrícola.

5. ABP vs OTAA: Métodos de activación en LoRaWAN

Para que un dispositivo transmita en una red LoRaWAN, necesita activarse. Hay dos métodos:

OTAA — Over-The-Air Activation (Activación por el aire)

El dispositivo hace un "join request" a la red. El servidor responde con un "join accept" que contiene las claves de sesión. Es más seguro (las claves se rotan) pero requiere que el dispositivo esté dentro del rango de un gateway en el momento de activación.

ABP — Activation By Personalization (Activación por personalización)

Las claves se pre-programan manualmente en el dispositivo y en el servidor. No hay handshake inicial. El dispositivo simplemente empieza a transmitir. Ventajas:

  • No depende de cobertura en el momento de instalación.
  • Ideal para dispositivos que se despliegan en campo y se encienden después.
  • Menor overhead de red (no consume un join procedure).

Desventaja: las claves no rotan, por lo que si se comprometen, hay que re-programar.

Nuestro código usa ABP con tres valores:

ClaveValor en el códigoFunción
devAddr 00d84a8c Dirección del dispositivo en la red (4 bytes). Como una "IP" pero para LoRaWAN.
nwkSKey be2954f4... Network Session Key: encripta y firma la comunicación entre dispositivo y red. El gateway y ChirpStack la usan para verificar integridad.
appSKey 0da02b24... Application Session Key: encripta el payload de extremo a extremo. Solo el dispositivo y la aplicación (ChirpStack) pueden leer los datos. Ni siquiera el operador de red puede verlos.
⚠️ Seguridad En un proyecto real, NUNCA publiques estas claves en GitHub ni en código abierto. Las de este archivo son para un entorno controlado de CENGICANA. En producción, las claves se almacenan en un archivo separado o en una zona segura del microcontrolador.

#define RAIN_PIN 9

Tres constantes fundamentales:

🔹 Bloque 6 — Configuración del módulo RFM95 (líneas 30-41)

const sRFM_pins RFM_pins = {
  .CS = SS,
  .RST = RFM_RST,
  .DIO0 = RFM_DIO0,
  .DIO1 = RFM_DIO1,
  .DIO2 = RFM_DIO2,
  .DIO5 = RFM_DIO5,
};

char outStr[255];
byte recvStatus = 0;

sRFM_pins es una estructura (struct) que mapea los pines del microcontrolador a las funciones del chip RFM95:

  • CS (Chip Select): Pin SS (Slave Select) del bus SPI. Activa el chip para comunicación.
  • RST (Reset): Pin de reinicio del módulo.
  • DIO0-DIO5: Pines de interrupción digital. El RFM95 los usa para señalar eventos: "transmisión completada" (DIO0), "paquete recibido" (DIO1), etc. La librería los configura automáticamente.

outStr[255] es un buffer para recibir downlinks (mensajes del servidor). recvStatus indica si se recibió algo (0 = nada, >0 = bytes recibidos).

🔹 Bloque 7 — setup() (líneas 43-71)

La función setup() se ejecuta UNA sola vez al encender el dispositivo.

1

Inicializar comunicación Serial

Serial.begin(115200);
delay(3000);

Abre el puerto serial USB a 115,200 baudios (bits por segundo). Los 3 segundos de delay dan tiempo a que el monitor serial del IDE de Arduino se conecte antes de que el programa empiece a enviar datos. Sin este delay, los primeros mensajes se perderían.

2

Configurar el pin del pluviómetro

pinMode(RAIN_PIN, INPUT_PULLUP);
delay(100);
estadoAnterior = digitalRead(RAIN_PIN);

INPUT_PULLUP activa una resistencia interna de ~20-50 kΩ que conecta el pin a VCC (3.3V). Esto garantiza que el pin lea HIGH cuando el reed switch está abierto. Cuando el switch se cierra (por el imán del balancín), conecta el pin a GND y lee LOW.

El delay(100) deja que la electrónica se estabilice. Luego leemos el estado inicial para que la primera iteración del loop no detecte un falso flanco.

3

Inicializar el módulo LoRa

if (!lora.init()) {
  Serial.println("ERROR: LoRa no detectado");
  while (1) {
  }
}

lora.init() verifica que el chip RFM95 responda en el bus SPI. Si falla (módulo mal conectado, dañado, o sin alimentación), el programa entra en un loop infinito (while(1){}). Esto es un "halt and catch fire" controlado: mejor detenerse que transmitir datos basura.

4

Configurar parámetros LoRaWAN

lora.setDeviceClass(CLASS_A);
lora.setDataRate(SF8BW125);
lora.setChannel(MULTI);
lora.setNwkSKey(nwkSKey);
lora.setAppSKey(appSKey);
lora.setDevAddr(devAddr);

Seis configuraciones en secuencia:

  1. CLASS_A: Modo de bajo consumo. Transmite, abre 2 ventanas RX, duerme.
  2. SF8BW125: Spreading Factor 8, ancho de banda 125 kHz.
  3. MULTI: El dispositivo puede saltar entre múltiples canales de frecuencia (frequency hopping). Esto es obligatorio en LoRaWAN para cumplir con regulaciones de espectro.
  4. setNwkSKey / setAppSKey / setDevAddr: Las credenciales ABP que definimos arriba.

Banner de inicio

🔹 Bloque 8 — loop() (líneas 73-146)

loop() se ejecuta una y otra vez, miles de veces por segundo. Es el "latido" del Arduino. Se divide en TRES partes:

Parte A — Detección de pulsos del pluviómetro (líneas 74-93)

int estadoActual = digitalRead(RAIN_PIN);

if (estadoAnterior == HIGH && estadoActual == LOW) {
  unsigned long ahora = millis();

  if ((ahora - ultimoPulso) > DEBOUNCE_MS) {
    ultimoPulso = ahora;
    pulsos++;

    float mmActual = pulsos * 0.2f;
    Serial.print("  [PULSO] Min:");
    Serial.print(minuto);
    Serial.print(" | Acumulado: ");
    Serial.print(pulsos);
    Serial.print(" pulsos = ");
    Serial.print(mmActual, 1);
    Serial.println(" mm");
  }
}
estadoAnterior = estadoActual;

Esta sección implementa un detector de flanco de bajada con debounce:

1
Leer el pin: digitalRead(RAIN_PIN) devuelve HIGH o LOW.
2
Detectar flanco: estadoAnterior == HIGH && estadoActual == LOW. Esto solo es verdad cuando el pin ACABA de cambiar de HIGH a LOW. Si ya estaba LOW y sigue LOW, no se dispara. Esto evita contar el mismo pulso miles de veces mientras el reed switch permanece cerrado.
3
Debounce: (ahora - ultimoPulso) > DEBOUNCE_MS. Un reed switch mecánico "rebota" — el contacto abre y cierra varias veces en milisegundos antes de estabilizarse. Sin debounce, un solo basculazo del balancín se contaría como 5-10 pulsos. Con 200 ms de ventana, ignoramos esos rebotes.
4
Registrar y mostrar: Si pasa ambas validaciones, incrementamos pulsos, calculamos mmActual = pulsos * 0.2, y lo imprimimos por serial.

Parte B — Transmisión periódica (líneas 95-143)

if (millis() - ultimoReporte >= INTERVALO_MS) {
  ultimoReporte = millis();

  float mm = pulsos * 0.2f;

  Serial.println("----------------------------------------");
  Serial.print("Minuto #");
  Serial.print(minuto);
  Serial.print(" | Pulsos: ");
  Serial.print(pulsos);
  Serial.print(" | Lluvia: ");
  Serial.print(mm, 1);
  Serial.println(" mm");

Cada 60 segundos, este bloque se dispara. La condición usa millis() — un reloj interno que cuenta milisegundos desde el arranque. Esto es no bloqueante: el loop sigue corriendo y detectando pulsos mientras espera. Si usáramos delay(60000), perderíamos todos los pulsos durante ese minuto.

Parte C — Construcción y envío del payload (líneas 109-137)

1

Convertir pulsos a unidades ChirpStack

uint16_t valor = (uint16_t)(pulsos * 2);
byte payload[2];
payload[0] = (valor >> 8) & 0xFF;
payload[1] = valor & 0xFF;

La conversión crítica: ChirpStack espera que cada unidad del payload represente 0.1 mm de lluvia. Como cada pulso del Davis son 0.2 mm, multiplicamos por 2. Si cayeron 5 pulsos → 5 × 0.2 = 1.0 mm → 5 × 2 = 10 unidades de 0.1 mm.

Big-endian: El byte más significativo (MSB) va primero. payload[0] = bits 8-15, payload[1] = bits 0-7. Ejemplo: valor = 10 (0x000A) → payload = [0x00, 0x0A]. Valor = 300 (0x012C) → payload = [0x01, 0x2C].

2

Mostrar el payload en hexadecimal

Serial.print("Enviando payload (0.1mm): 0x");
if (payload[0] < 0x10) {
  Serial.print("0");
}
Serial.print(payload[0], HEX);
if (payload[1] < 0x10) {
  Serial.print("0");
}
Serial.print(payload[1], HEX);
Serial.print(" = ");
Serial.print(valor);
Serial.println(" unidades");

Imprime el payload en formato legible. Los if agregan un cero a la izquierda cuando el byte es menor a 0x10, para que siempre se vean 2 dígitos hex por byte. Ejemplo: 0x000A = 10 unidades.

3

Transmitir por LoRaWAN

lora.sendUplink((char *)payload, 2, 0, 2);
Serial.println("Uplink encolado correctamente");

sendUplink encola el payload para transmisión. Parámetros:

  • (char *)payload: El array de 2 bytes convertido a puntero char.
  • 2: Longitud del payload en bytes.
  • 0: Puerto 0 (puerto por defecto en LoRaWAN, sin significado especial).
  • 2: Solicitar confirmación (confirmed uplink). El servidor debe responder con un ACK. Si no llega, la librería retransmite automáticamente.

La transmisión real no ocurre aquí — solo se encola. El lora.update() al final del loop es quien realmente procesa la cola y transmite cuando las ventanas de CLASS A lo permiten.

4

Recibir downlink (respuesta del servidor)

recvStatus = lora.readData(outStr);
if (recvStatus) {
  Serial.print("Downlink recibido: ");
  Serial.println(outStr);
}

Después de transmitir, CLASS A abre dos ventanas de recepción (RX1 y RX2). Si ChirpStack envió un downlink (por ejemplo, un comando de reconfiguración), readData lo captura y lo imprime.

5

Resetear contadores para el siguiente minuto

pulsos = 0;
minuto = (minuto + 1) % 1440;

pulsos = 0 resetea el acumulador para el próximo intervalo. minuto = (minuto + 1) % 1440 incrementa el contador de minutos, y con módulo 1440 (24 horas × 60 minutos) lo hace circular: después de 1439 vuelve a 0. Esto es solo para logging; no afecta los datos transmitidos.

Parte D — Mantenimiento de LoRa (línea 145)

lora.update();

Esta línea es crítica y se ejecuta en CADA iteración del loop, no solo cuando se transmite. lora.update() es la máquina de estados interna de la librería LoRaWAN. Se encarga de:

  • Procesar la cola de uplinks pendientes.
  • Iniciar transmisión cuando llega el momento (respetando duty cycle y regulaciones).
  • Abrir ventanas RX1 y RX2 después de cada uplink.
  • Manejar retransmisiones si no llega ACK.
  • Procesar downlinks entrantes.
  • Gestionar el canal MULTI (frequency hopping).

Sin esta llamada constante, el módulo LoRa simplemente no haría nada. Es el "motor" que mantiene viva la comunicación.

7. El Payload: Cómo se empaquetan los datos

La regla de conversión

📐 Fórmula sagrada
valor_payload = pulsos × 2
lluvia_mm = valor_payload × 0.1
O directamente: lluvia_mm = pulsos × 0.2

Tabla de ejemplos

PulsosLluvia realvalor (payload)Bytes (big-endian hex)
00.0 mm00x00 0x00
10.2 mm20x00 0x02
51.0 mm100x00 0x0A
255.0 mm500x00 0x32
12825.6 mm2560x01 0x00
500100.0 mm10000x03 0xE8
163833276.6 mm327660x7F 0xFE

Con 2 bytes (16 bits), el valor máximo es 65535, equivalente a 6553.5 mm de lluvia en un minuto — físicamente imposible, así que nunca habrá overflow.

Big-Endian vs Little-Endian

Big-endian significa que el byte más significativo (el de mayor peso) se transmite PRIMERO. Es el estándar de red (TCP/IP, LoRaWAN). Ejemplo con valor 300 (0x012C):

FormatoByte 0Byte 1
Big-endian (nuestro código)0x01 (MSB)0x2C (LSB)
Little-endian (x86, Arduino nativo)0x2C (LSB)0x01 (MSB)

El código de ChirpStack (codec) debe saber que el payload es big-endian para reconstruir el valor correctamente: valor = payload[0] × 256 + payload[1].

8. Timing y Debounce: El arte de medir sin errores

¿Por qué millis() y no delay()?

delay(60000) CONGELA el microcontrolador por 60 segundos. Durante ese tiempo:

  • ❌ No se lee el pin del pluviómetro → se pierden pulsos.
  • ❌ No se ejecuta lora.update() → se pierden downlinks y ACKs.
  • ❌ No se puede hacer NADA más.

millis() es un contador que avanza solo, en hardware. Leerlo toma microsegundos. El patrón if (millis() - ultimoReporte >= INTERVALO_MS) permite que el loop siga corriendo libremente y solo actúe cuando se cumple el tiempo. Esto se llama programación no bloqueante y es fundamental en IoT.

El problema del rebote (bounce)

Un reed switch es un interruptor mecánico: dos láminas metálicas que se tocan cuando un imán se acerca. Pero no hacen contacto limpio — vibran, rebotan, generan múltiples aperturas y cierres en milisegundos:

Cómo funciona el debounce en nuestro código

if ((ahora - ultimoPulso) > DEBOUNCE_MS) {  // 200 ms
  ultimoPulso = ahora;                          // actualizar timestamp
  pulsos++;                                     // contar UN solo pulso
}

Después de aceptar un pulso, el código actualiza ultimoPulso al tiempo actual. Cualquier flanco que llegue en los siguientes 200 ms será rechazado porque (ahora - ultimoPulso) será menor que 200. Para cuando pasan los 200 ms, los rebotes ya terminaron y el switch está estable (firmemente cerrado o firmemente abierto).

💡 ¿Por qué 200 ms? Un tipping bucket típico tarda entre 100 y 300 ms en bascular completamente. Con 200 ms cubrimos el rebote sin riesgo de perder dos basculazos reales consecutivos (que en la práctica nunca ocurren a menos de 500 ms de separación incluso en tormenta torrencial).

Overflow de millis()

millis() retorna un unsigned long (32 bits). Su valor máximo es 4,294,967,295 ms ≈ 49.7 días. Después de eso, se desborda a 0. Pero nuestro código es inmune a este overflow gracias a la aritmética sin signo: 

// Justo antes del overflow:
millis() = 4294967290
ultimoReporte = 4294960000
millis() - ultimoReporte = 7290  // OK, positivo

// Justo después del overflow (millis vuelve a 0):
millis() = 500
ultimoReporte = 4294960000
millis() - ultimoReporte = 500 - 4294960000
  = 500 + (4294967295 - 4294960000 + 1)  // aritmética modular sin signo
  = 500 + 7296 = 7796  // sigue funcionando correctamente

Esta es una propiedad mágica de la resta sin signo en C/C++. Mientras el intervalo sea menor a 49.7 días (y 60 segundos ciertamente lo es), el código funciona para siempre.

9. Glosario IoT para estudiantes

Términos esenciales que aparecen en este proyecto y en el mundo IoT en general:

LoRa (Long Range)
Técnica de modulación de radio por chirp spread spectrum (CSS). Largo alcance (km), bajo consumo (mW), baja velocidad (kbps). Capa FÍSICA del stack.
LoRaWAN
Protocolo de red MAC sobre LoRa. Define autenticación, encriptación (AES-128), clases de dispositivos (A/B/C), y arquitectura estrella (dispositivos → gateway → network server).
ABP (Activation By Personalization)
Método de activación donde DevAddr, NwkSKey y AppSKey se pre-programan manualmente. Sin handshake. Ideal para despliegues en campo sin cobertura inicial.
OTAA (Over-The-Air Activation)
Método de activación con join request/accept. Las claves de sesión se derivan dinámicamente. Más seguro pero requiere cobertura al encender.
DevAddr
Dirección del dispositivo en la red LoRaWAN (32 bits). Identifica al dispositivo de forma única dentro de la red.
NwkSKey (Network Session Key)
Clave AES-128 que protege la integridad del mensaje entre dispositivo y network server. Verifica que el paquete no fue alterado en tránsito.
AppSKey (Application Session Key)
Clave AES-128 que encripta el payload de extremo a extremo. Solo el dispositivo y el application server pueden descifrar los datos.
Spreading Factor (SF)
Número de chips por símbolo en modulación LoRa. SF7=128 chips (rápido, menor alcance), SF12=4096 chips (lento, máximo alcance). Cada incremento de SF duplica el alcance y el tiempo de transmisión.
Gateway LoRaWAN
Antena + concentrador que recibe transmisiones LoRa de múltiples dispositivos y las reenvía al network server por Internet (TCP/IP). No decodifica payloads, solo reenvía.
ChirpStack
Network server LoRaWAN de código abierto. Recibe paquetes de los gateways, verifica integridad (NwkSKey), decodifica payloads (AppSKey + codec), y expone los datos vía API REST, MQTT, o integraciones.
Uplink
Mensaje que va del dispositivo hacia el network server. En CLASS A, el dispositivo inicia la comunicación.
Downlink
Mensaje que va del network server hacia el dispositivo. En CLASS A, solo es posible en las ventanas RX1/RX2 que se abren justo después de un uplink.
Payload
Los datos útiles (carga útil) del mensaje. En nuestro caso, 2 bytes con la lluvia acumulada en unidades de 0.1 mm. El payload NO incluye headers MAC ni metadata de red.
Big-Endian
Orden de bytes donde el MSB (byte más significativo) va primero. Estándar de red. Ej: 0x012C se transmite como [0x01, 0x2C].
Debounce
Técnica para filtrar rebotes mecánicos de interruptores. Ignora transiciones que ocurren dentro de una ventana de tiempo después de la primera transición válida.
INPUT_PULLUP
Modo de pin en Arduino que activa una resistencia interna conectada a VCC. El pin lee HIGH por defecto; se va a LOW cuando un switch externo lo conecta a GND.
millis()
Función de Arduino que retorna milisegundos desde el arranque. Base de la programación no bloqueante. Se desborda cada ~49.7 días pero la aritmética sin signo lo maneja correctamente.
Tipping Bucket
Mecanismo de medición de lluvia: un balancín con dos cubos calibrados. Cada basculazo representa una cantidad fija de lluvia (0.2 mm en el Davis 7345.440).
Reed Switch
Interruptor magnético: dos láminas metálicas selladas en vidrio que se tocan en presencia de un campo magnético. Sin contacto físico externo, ideal para ambientes húmedos.
SPI (Serial Peripheral Interface)
Bus de comunicación serial síncrono entre microcontrolador y periféricos. El BastWAN usa SPI para comunicarse con el chip RFM95 (LoRa).
Duty Cycle
Límite regulatorio de tiempo de transmisión. En banda 868 MHz (EU), un dispositivo solo puede transmitir el 1% del tiempo. La librería LoRaWAN lo respeta automáticamente.
Codec (ChirpStack)
Función JavaScript en ChirpStack que decodifica el payload binario a valores legibles (ej: bytes → mm de lluvia). Se configura por dispositivo.
